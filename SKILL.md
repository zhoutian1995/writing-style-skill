# 周老板写作风格 Skill v2

**平台 × 风格 = 你的 Profile。内置自动学习，越写越像你。**

---

## 🚨 写作前决策树（必读）

```
第一步：确认平台（硬约束）
├── 公众号 → 字数800-2000，markdown，封面图第一行
├── 推特 → 280字/条，纯文本，不渲染markdown
├── 小红书 → 300-800字，emoji清单，标签5-10
└── 博客 → 2000+字，标准markdown，可加TOC

第二步：确认风格（软约束）
├── 技术实践 → 踩坑叙事，第一人称
├── 观点评论 → 趋势判断，有锋利立场
├── 干货攻略 → 工具推荐/教程，第二人称
└── 深度研究 → 长文分析，第三人称

第三步：加载 Profile = 平台 × 风格
└── 例如：公众号 × 观点评论 = 今天这篇文章

写完之后：
├── 记录原稿 → python3 scripts/observe.py record-original <file>
├── 人类确认后 → python3 scripts/observe.py record-final <file> --match <hash>
└── 积累够了 → python3 scripts/improve.py auto --skill .
```

## 【1】平台规范（第一层：硬约束）

### 公众号

| 项目 | 规范 |
|------|------|
| 字数 | 800-2000字 |
| 格式 | markdown，表格、代码块 |
| 段落 | 2-4行，手机阅读友好 |
| 封面图 | ⚠️ **必须第一行！** 赛博朋克风，900x383px，AI生成背景+PIL叠加标题 |
| 结构 | 开头3句定生死 → 中间逻辑线 → 结尾金句或互动钩子 |
| 互动 | 引导点赞/在看/转发，但要自然不生硬 |

### 推特

| 项目 | 规范 |
|------|------|
| 字数 | 280字/条 |
| 格式 | 纯文本（X不渲染markdown） |
| 结构 | 一条推文 = 一个观点；长文用thread拆，每条独立可读 |
| 钩子 | 前10个字决定看不看，开头必须抓人 |
| Thread | 每条可独立传播，不用"1/7"这种序号 |

### 小红书

| 项目 | 规范 |
|------|------|
| 字数 | 300-800字 |
| 格式 | emoji清单体，不用markdown |
| 标题 | 20字以内，关键词前置，带emoji |
| 结构 | 开头亮点/痛点 → 中间干货密集 → 结尾互动引导 |
| 风格 | 像朋友在分享，不像老师在讲课 |
| 标签 | 5-10个相关标签 |

### 博客

| 项目 | 规范 |
|------|------|
| 字数 | 2000+字 |
| 格式 | 标准markdown，可加TOC目录 |
| 代码 | 完整示例（不是伪代码） |
| 风格 | 正式但不学术，像技术文档 |
| 结构 | 可长篇论述，逻辑严密 |

## 【2】风格规范（第二层：软约束）

### 风格A：技术实践（踩坑叙事）

- **人称**：第一人称
- **结构**：场景开头 → 问题 → 探索 → 解决 → 反思
- **情绪**：荒唐感、成就感、"原来如此"
- **核心**：分享真实踩坑经历，比写教程有说服力10倍

### 风格B：观点评论（趋势判断）

- **人称**：第一人称，但更锋利
- **结构**：争议事件 → 我怎么看 → 为什么这么看 → 对你有什么用
- **情绪**：意外、共鸣、"我不同意"
- **核心**：有明确立场，不说"各有优劣"这种废话
- **注意**：不要总引用别人的话，用自己的话说

### 风格C：干货攻略（工具推荐/教程）

- **人称**：第二人称为主（"你可以这样做"）
- **结构**：痛点 → 方案 → 步骤 → 效果
- **情绪**：实用感、"学到了"
- **核心**：具体可执行，不说"建议关注"这种空话

### 风格D：深度研究（长文分析）

- **人称**：第三人称为主
- **结构**：背景 → 现状 → 分析 → 判断 → 行动建议
- **情绪**：专业感、权威感
- **核心**：数据驱动，有图表更佳

## 【5】核心写作规则（全平台共享，永远不变）

### 禁止词

<!-- 自动学习会在这里积累更多 -->
- 值得注意的是
- 综上所述
- 本文将介绍
- 深入探讨
- 让我们来看一下
- 总的来说
- 在当今时代

### 句式偏好

- ✅ 结论前置（先说结果，再说过程）
- ✅ 场景开头（凌晨两点，屏幕突然报错）
- ✅ 短句为主（一个句子不超过 20 字）
- ✅ 数据 > 形容词（"3800元"不是"便宜"，"22 tok/s"不是"快"）
- ❌ 不要学术导语（"本文将探讨..."）
- ❌ 不要被动句（"被广泛认为..."）
- ❌ 不要排比三连（"不仅...而且...更是..."）

### 平台适配矩阵

| 平台 × 风格 | 技术实践 | 观点评论 | 干货攻略 | 深度研究 |
|-------------|---------|---------|---------|---------|
| **公众号** | ✅ 主力 | ✅ 主力 | ✅ 可用 | ⚠️ 精简到2000字 |
| **推特** | ✅ thread | ✅ 一条thread | ✅ 一条推文 | ❌ 不适合 |
| **小红书** | ⚠️ 可精简 | ✅ 3个要点 | ✅ 主力 | ❌ 不适合 |
| **博客** | ✅ 主力 | ✅ 可用 | ✅ 可用 | ✅ 主力 |

> ✅ = 适合  ⚠️ = 需要调整  ❌ = 不推荐

## 【6】爆款文章拆解（13维度分析法）

**分析爆款文章时，按以下维度拆解：**

| # | 维度 | 分析要点 |
|---|------|---------|
| 1 | 核心观点 | 一句话概括文章最想表达的 |
| 2 | 副观点 | 支撑核心观点的子论点 |
| 3 | 说服策略 | 案例论证/数据支持/权威背书/反转对比 |
| 4 | 情绪触发点 | 哪些地方让读者共鸣/焦虑/渴望 |
| 5 | 金句 | 值得摘抄、可复用的句子 |
| 6 | 情感曲线 | 开头情绪→中段起伏→结尾收束 |
| 7 | 情感层次 | 表层→深层→潜意识 |
| 8 | 论证方式 | 多种交替使用，避免单一 |
| 9 | 视角转化 | 第一人称/第二人称/第三人称切换 |
| 10 | 语言风格 | 语气/句式/词汇特征 |
| 11 | 情绪价值句式 | 鼓励型/认可型/启发型/治愈型 |
| 12 | 刺痛句式 | 揭露真相型/打破幻想型/痛点放大型/警告型 |
| 13 | 互动钩子 | 提问型/填空型/争议型/晒成果型/反转悬念型 |

**互动钩子类型：**

| 类型 | 示例 | 效果 |
|------|------|------|
| 提问型 | "你觉得 Claude 还是 GPT 更适合做 Agent？" | 用户看到就想回答 |
| 填空型 | "我用 AI 最崩溃的一次是 ____" | 开放式，每个人都有话说 |
| 争议型 | "Cursor 靠 Kimi 做底座算不算抄袭？" | 站队就有流量 |
| 晒成果型 | "你用 AI 做过最酷的东西是什么？" | 用户想炫耀 |
| 反转悬念型 | "但是后来我发现了一个问题……" | 不说完，让人追问 |

## 【7】公众号全流程（公众号专用）

**写公众号文章必须按这个流程走，不能跳步。**

### 第一步：选题 + 资料搜集
- 确定核心观点（一句话能说清）
- 搜集 3+ 个信息源交叉验证
- 提炼"为什么重要"和"对读者的启发"

### 第二步：生成封面图
- **风格**：赛博朋克/科技风（深色背景+发光线条+电路板元素）
- **尺寸**：900x383px
- **流程**：AI生成高质量背景图（不要文字）→ PIL叠加中文标题
- **⚠️ 铁律**：封面图必须放文章**第一行**

### 第三步：写正文
- 按选定的风格Profile写初稿
- 800-2000字
- 结尾有金句或互动钩子

### 第四步：排版检查
- [ ] 封面图在第一行？
- [ ] markdown格式正确？
- [ ] 段落2-4行，没有超长段落？
- [ ] 有错别字/语病？
- [ ] 图片和文字内容匹配？

### 第五步：推送确认
- **铁律**：推送前必须把文章内容发给周老板确认
- 确认无误后再推送，绝不自己决定

## 【7.5】配图生成流程（公众号专用）

**封面图和正文配图的生成流程，必须按以下步骤走。**

### 工具选择
- **背景图生成**：用 `cli-switch --tool gemini nanobanana` 切换到 nano banana2（gemini-3.1-flash-image-preview），然后用 Gemini CLI 或直接调 API 生成
- **中文标题叠加**：用 PIL（Python）在背景图上叠加清晰中文标题
- **⚠️ 禁止**：不要让 AI 直接生成带文字的图片（拼写错误率高）

### 封面图规范
- **尺寸**：900x383px（公众号封面比例 2.35:1）
- **风格**：赛博朋克/科技风（深色背景 + 发光线条 + 电路板元素）
- **底部**：加渐变遮罩（黑色半透明），让白色标题清晰可读
- **标题**：主标题白色+发光描边，副标题青色
- **⚠️ 铁律**：封面图必须放在文章**第一行**

### 生成流程
```bash
# 1. 切换到 nano banana2
cli-switch --tool gemini nanobanana

# 2. 生成背景图（用API或CLI，不要含文字）
curl -s "https://generativelanguage.googleapis.com/v1beta/models/gemini-3.1-flash-image-preview:generateContent?key=$GEMINI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"contents":[{"parts":[{"text":"Generate an image: [prompt描述，末尾加 no text no words no letters]"}]}],"generationConfig":{"responseModalities":["TEXT","IMAGE"]}}'

# 3. 用PIL叠加中文标题
python3 -c "
from PIL import Image, ImageDraw, ImageFont
# 调整尺寸 → 加渐变遮罩 → 加标题文字 → 保存
"

# 4. 自审图片质量
# 用 image 工具检查：清晰度、可读性、风格、整体质量
```

### 正文配图
- 每 2-3 段插一张，打破视觉疲劳
- 主题：概念图解（如 Harness 三根支柱）、数据对比图、场景插图
- 同样用 nano banana2 生成背景 + PIL 叠加
- **风格**：简约扁平插画风（不是赛博朋克），浅色背景，柔和色调
- **封面图**才是赛博朋克风格，正文插图是简约插画风
- 生成后飞书发给周老板审核，确认后再插入文章

## 【8】写后自审（AI 自我评价）

**写完文章后，必须从 6 个维度给自己打分（1-10），并给出具体优化建议。不要锁死规则，要主动提意见。**

| 维度 | 评价标准 |
|------|---------|
| **内容深度** | 核心概念解释清楚了吗？读者看完能复述吗？ |
| **字数匹配** | 内容需要多少字才说得清楚？不要为了简洁牺牲深度，也不要注水 |
| **情绪触发** | 读者在哪些位置会产生共鸣/意外/焦虑/启发？ |
| **案例质量** | 案例够具体吗？是内部经历还是用户能共鸣的通用场景？ |
| **金句密度** | 有没有让读者想转发的句子？够不够？ |
| **互动钩子** | 结尾让人想回复吗？ |

**自审规则：**
- 字数是指导，不是死线。Harness 这种需要展开的概念，2500 字比 1500 字好
- 如果自审发现某个维度低于 6 分，主动提出优化建议
- 不要等用户问"你觉得写得怎么样"，写完就自审
- 主动说问题在哪，不要只报喜

## 🔄 自动学习（内置）

**这个 skill 会从你的修改中自动学习。不需要手动写规则。**

### 工作原理

```
AI 用这个 skill 写初稿
    ↓
你改到满意
    ↓
脚本 diff 两版 → 提取你改了什么
    ↓
新规则自动写入这个 SKILL.md
    ↓
下次 AI 写出来就更像你
```

### 只需要两个数据点

- **original**: AI 生成的第一版
- **final**: 你最终确认的版本

中间改了几轮不管。只比较首尾。

### Agent 操作指南

**写完内容后：**
```bash
python3 scripts/observe.py record-original <file> --account <账号> --content-type <类型>
```

**人类确认最终版后：**
```bash
python3 scripts/observe.py record-final <file> --match <hash>
```

**提取规则（手动或 cron 自动）：**
```bash
python3 scripts/improve.py auto --skill .
```

### 规则分级

| 级别 | 含义 | 处理方式 |
|------|------|---------|
| P0 | 高置信度（多次出现） | 自动应用 |
| P1 | 中置信度 | 人工确认 |
| P2 | 低置信度（仅 1 次） | 存档观察 |

### 安全

- 每次更新前自动备份 SKILL.md
- `improve.py rollback` 一键回滚
- auto 模式只应用 P0

## 📊 CLI 参考

### observe.py（零依赖纯 Python）

| 命令 | 功能 |
|------|------|
| `record-original <file>` | 记录 AI 原稿 |
| `record-final <file> --match <hash>` | 记录最终版 |
| `pending` | 查看待配对 |
| `stats` | 统计 |

### improve.py（需要 LLM CLI）

| 命令 | 功能 |
|------|------|
| `extract [--days 7]` | 提取改进建议 |
| `auto` | 提取 + 自动应用 P0 |
| `show` | 查看提案 |
| `apply <id>` | 应用提案 |
| `rollback` | 回滚 |

支持的 LLM CLI: `claude` / `llm` / `IMPROVE_LLM_CMD` 环境变量
