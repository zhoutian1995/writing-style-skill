## 【0】Voice Dimensions（量化你的风格）

**用 1-10 分定义你的风格维度。AI 比"写得自然一点"这种话更容易理解数字。**

| Dimension | Score | 说明 |
|-----------|-------|------|
| **formal_casual** | **8/10** | 偏随意，用"我"说话，像朋友聊天 |
| **technical_accessible** | **7/10** | 有技术深度但不晦涩，外行也能看懂70% |
| **serious_playful** | **6/10** | 可以幽默但不失专业，偶尔冷幽默 |
| **concise_elaborate** | **4/10** | 偏简洁，不超1500字，不废话 |
| **reserved_expressive** | **8/10** | 直接、有态度、有观点，不中立 |

> 💡 这些分数会随着你的修改自动校准。

## 【1.5】公众号全流程

**写公众号文章必须按这个流程走，不能跳步。**

### 第一步：选题 + 资料搜集
- 确定核心观点（一句话能说清）
- 搜集 3+ 个信息源交叉验证
- 提炼"为什么重要"和"对读者的启发"

### 第二步：生成封面图
- **风格**：赛博朋克/科技风（深色背景+发光线条+电路板元素）
- **尺寸**：900x383px（公众号封面比例）
- **流程**：用 AI 生成高质量背景图（不要文字）→ 用 PIL 叠加中文标题
- **⚠️ 铁律**：封面图必须放在文章**第一行**，参考"小龙虾诞生记.md"格式

### 第三步：写正文
- 按【2】写作规则写初稿
- 1000-1500字（偏简洁）
- 结尾有金句

### 第四步：排版检查
- [ ] 封面图在第一行？
- [ ] markdown表格/代码块格式正确？
- [ ] 段落2-4行，没有超长段落？
- [ ] 有错别字/语病？
- [ ] 图片和文字内容匹配？

### 第五步：推送确认
- **铁律**：推送前必须把文章内容发给周老板确认
- 确认无误后再推送，绝不自己决定

## 【1.6】爆款文章拆解（12维度分析法）

**分析爆款文章时，按以下维度拆解：**

| # | 维度 | 分析要点 |
|---|------|---------|
| 1 | 核心观点 | 一句话概括文章最想表达的 |
| 2 | 副观点 | 支撑核心观点的子论点 |
| 3 | 说服策略 | 案例论证/数据支持/权威背书/反转对比 |
| 4 | 情绪触发点 | 哪些地方让读者共鸣/焦虑/渴望 |
| 5 | 金句 | 值得摘抄、可复用的句子 |
| 6 | 情感曲线 | 开头情绪→中段起伏→结尾收束 |
| 7 | 情感层次 | 表层讲了什么→深层暗示了什么→潜意识触动了什么 |
| 8 | 论证方式 | 用了几种论证方式？单一还是丰富？ |
| 9 | 视角转化 | 第一人称/第三人称/是否有视角切换 |
| 10 | 语言风格 | 语气/句式/词汇特征 |
| 11 | 情绪价值句式 | 鼓励型/认可型/启发型/治愈型 |
| 12 | 刺痛句式 | 揭露真相型/打破幻想型/痛点放大型/警告型 |
| 13 | 互动钩子 | 提问型/填空型/争议型/晒成果型/反转悬念型 |

**互动钩子详解**（爆款帖的隐藏维度）：

| 类型 | 示例 | 效果 |
|------|------|------|
| 提问型 | "你觉得 Claude 还是 GPT 更适合做 Agent？" | 用户看到就想回答 |
| 填空型 | "我用 AI 最崩溃的一次是 ____" | 开放式，每个人都有话说 |
| 争议型 | "Cursor 靠 Kimi 做底座算不算抄袭？" | 站队就有流量 |
| 晒成果型 | "你用 AI 做过最酷的东西是什么？" | 用户想炫耀，评论区自然活 |
| 反转悬念型 | "但是后来我发现了一个问题……" | 不说完，让人追问 |

**使用格式**：
```
请用爆款文章拆解12维度分析以下文章：
[粘贴文章内容]
```

## 【2】写作规则

### 核心规则

1. **第一人称叙事** — 用"我"讲故事，不是写报告
2. **踩坑 > 教程** — 分享真实踩坑经历，比写教程有说服力 10 倍
3. **数据 > 形容词** — "便宜"不行，要说"3800元"；"快"不行，要说"22 tok/s"
4. **段落 2-4 行** — 不写长段落，超过 4 行就拆
5. **结尾有金句** — 从技术上升到思考，一句话收尾
6. **诚实 > 营销** — 主动说局限性，不吹不黑

### 禁止词

<!-- 自动学习会在这里积累更多 -->
- 值得注意的是
- 综上所述
- 本文将介绍
- 深入探讨
- 让我们来看一下
- 总的来说
- 在当今时代

### 句式偏好

- ✅ 结论前置（先说结果，再说过程）
- ✅ 场景开头（凌晨两点，屏幕突然报错）
- ✅ 短句为主（一个句子不超过 20 字）
- ❌ 不要学术导语（"本文将探讨..."）
- ❌ 不要被动句（"被广泛认为..."）
- ❌ 不要排比三连（"不仅...而且...更是..."）

> 💡 **不需要一开始就写完。** 这些规则会通过你的修改自动积累。
> 跑完 10 次写作→修改循环后，这里会长出几十条精准规则。

## 【3】格式规范

### 平台适配

| 平台 | 格式要求 | 字数 | 特殊规则 |
|------|---------|------|---------|
| 公众号 | 封面图第一行、markdown表格、代码块 | 1000-1500字 | ⚠️ 封面图必须第一行！图片用AI生成背景+PIL叠加标题 |
| 小红书 | emoji多、分段短、不要markdown、口语化 | 500-800字 | 每段不超过3行，多用emoji分隔 |
| 博客 | 标准markdown、可长文、技术深度更高 | 2000+字 | 可以加目录、代码示例 |
| X/Twitter | 纯文本、不渲染markdown | 280字限制 | 一条推文一个观点，用换行分隔 |

## 🔄 自动学习（内置）

**这个 skill 会从你的修改中自动学习。不需要手动写规则。**

### 工作原理

```
AI 用这个 skill 写初稿
    ↓
你改到满意
    ↓
脚本 diff 两版 → 提取你改了什么
    ↓
新规则自动写入这个 SKILL.md
    ↓
下次 AI 写出来就更像你
```

### 只需要两个数据点

- **original**: AI 生成的第一版
- **final**: 你最终确认的版本

中间改了几轮不管。在 Google Doc 里来回改了 10 次？无所谓，只比较首尾。

### Agent 操作指南

**写完内容后：**
```bash
python3 scripts/observe.py record-original <file> --account <账号> --content-type <类型>
```

**人类确认最终版后：**
```bash
python3 scripts/observe.py record-final <file> --match <hash>
```

**提取规则（手动或 cron 自动）：**
```bash
python3 scripts/improve.py auto --skill .
```

### 规则分级

| 级别 | 含义 | 处理方式 |
|------|------|---------|
| P0 | 高置信度（多次出现） | 自动应用 |
| P1 | 中置信度 | 人工确认 |
| P2 | 低置信度（仅 1 次） | 存档观察 |

### 安全

- 每次更新前自动备份 SKILL.md
- `improve.py rollback` 一键回滚
- auto 模式只应用 P0

## 📊 CLI 参考

### observe.py（零依赖纯 Python）

| 命令 | 功能 |
|------|------|
| `record-original <file>` | 记录 AI 原稿 |
| `record-final <file> --match <hash>` | 记录最终版 |
| `pending` | 查看待配对 |
| `stats` | 统计 |

### improve.py（需要 LLM CLI）

| 命令 | 功能 |
|------|------|
| `extract [--days 7]` | 提取改进建议 |
| `auto` | 提取 + 自动应用 P0 |
| `show` | 查看提案 |
| `apply <id>` | 应用提案 |
| `rollback` | 回滚 |

支持的 LLM CLI: `claude`（Claude Code）/ `llm`（pip install llm）/ `IMPROVE_LLM_CMD` 环境变量
